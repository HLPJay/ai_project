# Lyrics Generation (Step ①)

## 🎯 目标

给定一个主题（theme），生成一首完整歌曲的歌词，输出：
- 歌词文件（含曲名、风格标签、歌词正文）
- 项目元数据更新（供后续步骤使用）

## 📦 边界

**输入：**
- `theme` — 用户提供的中文主题描述字符串

**本步骤产出：**
- `{project_dir}/audio/lyrics.txt`
- 更新 `{project_dir}/metadata/info.json`（song_title / style_tags / theme）

**本步骤不负责：**
- 音乐生成（Step②）
- 歌词与音频的对齐（Step③）
- 主题合法性校验（假设已通过用户输入验证）

## 📝 步骤

### Step 1: 调用 MiniMax /lyrics_generation API

**请求：**
```
POST https://api.minimaxi.com/v1/lyrics_generation
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `mode` | string | ✅ | 固定值：`write_full_song` |
| `prompt` | string | ✅ | 主题描述（中文） |

**响应：**
| 字段 | 类型 | 说明 |
|------|------|------|
| `song_title` | string | 自动生成的歌曲名 |
| `style_tags` | string | 逗号分隔的风格标签，如 `"nostalgic,warm,childhood"` |
| `lyrics` | string | 完整歌词，含段落标记 `[Verse]`, `[Chorus]` 等 |
| `base_resp.status_code` | int | `0` = 成功，非0 = 错误 |

### Step 2: 写入 lyrics.txt

输出文件路径：`{project_dir}/audio/lyrics.txt`

文件格式：
```
## <song_title>
## Tags: <style_tags>
## Theme: <theme>
## Generated by MiniMax Lyrics Generation API

[Verse 1]
...
[Chorus]
...
```

- `## `（两个 `#` + 空格）开头的是元数据行，区别于歌词正文中可能出现的单个 `#`
- 段落标记 `[Verse]`, `[Chorus]` 等保留在歌词文本中
- 歌词正文可能含特殊字符，不得被 shell 展开

### Step 3: 更新 metadata/info.json

更新字段：
```json
{
  "song_title": "<song_title>",
  "style_tags": ["nostalgic", "warm", "childhood"],
  "theme": "<theme>"
}
```

- `style_tags` 从逗号分隔字符串转为数组

## 📊 过程日志输出

本步骤在执行过程中向用户输出可读的进度日志，同时更新 `status.json`。

**日志级别：**

| 级别 | 格式 | 时机 |
|------|------|------|
| `🔄` | `[① lyrics] starting...` | 子步骤开始 |
| `🔄` | `[① lyrics] calling API...` | 正在调用 MiniMax API |
| `🔄` | `[① lyrics] API call failed, retry N/3 in Xs...` | API 失败重试中 |
| `🔄` | `[① lyrics] API call succeeded` | API 调用成功 |
| `🔄` | `[① lyrics] parsing response...` | 解析响应中 |
| `🔄` | `[① lyrics] writing lyrics.txt...` | 写入歌词文件 |
| `🔄` | `[① lyrics] updating metadata...` | 更新元数据 |
| `✅` | `[① lyrics] completed: title='...', tags='...'` | 成功完成 |
| `❌` | `[① lyrics] FAILED: <reason>` | 执行失败 |
| `⚠️` | `[① lyrics] interrupted: <reason>` | 被用户打断 |

**子步骤检查点：** 脚本在每个子步骤开始前调用 `check_interrupt()`，发现打断信号则输出 `⚠️` 并以退出码 0 终止。

**status.json 更新时机：**
- 子步骤开始 → `"status": "running"`
- 子步骤完成 → 保持 `"running"`
- 整个步骤完成 → `"status": "completed"`
- 失败 → `"status": "failed"`
- 打断 → `"status": "interrupted"`

**打断机制：** 详见 [SKILL.md — Unified Status Tracking](../SKILL.md#unified-status-tracking)。

## ⚙️ 其他要求

### 错误处理

| 错误类型 | 处理方式 |
|---------|---------|
| API 返回 `status_code != 0` | 输出错误信息到日志，脚本以非零退出码终止 |
| API 无响应 / 超时 | curl `--max-time 60`，超时则视为失败 |
| API 失败 | 重试 3 次，间隔 2/4/8 秒（指数退避） |
| 响应 JSON 解析失败 | 输出 "Invalid API response"，脚本终止 |
| 输入文件缺失 | 不适用（本步骤无前置文件依赖） |

### 边界条件

| 条件 | 说明 |
|------|------|
| `theme` 为空 | 脚本输出错误信息并以非零码退出 |
| `MINIMAX_TOKEN` 未设置 | 脚本输出错误信息并以非零码退出 |
| `song_title` 为空 | 使用默认值 `"Untitled"` |
| `style_tags` 为空 | 写入空数组 `[]` |
| 歌词内容含 `$` 等特殊字符 | 必须防止 shell 展开（不使用 heredoc展开变量） |

### 幂等性

本步骤是幂等的：
- 重复执行会覆盖已有的 `audio/lyrics.txt` 和 `info.json`
- 适用于从失败点恢复或重新生成

### 性能约束

- API 调用预期耗时：< 10 秒
- 整体步骤预期耗时：< 30 秒（含网络延迟和 1 次重试）
