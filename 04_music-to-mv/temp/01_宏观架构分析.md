# Music-to-MV 宏观架构分析

## 一、现状评估 (AS-IS)

### 1.1 现有架构类型
**架构模式：松散脚本链（Chained Scripts）**

```
入口点：scripts/produce_mv.sh 为核心驱动
数据载体：物理文件（.mp3, .srt, .png）
逻辑流：顺序调用 Python 脚本处理特定任务
```

### 1.2 优点
- **极致解耦**：每一个脚本（如 analyze_srt.py）都可以脱离主程序独立测试
- **符合 UNIX 哲学**：一个工具只做一件事，通过管道或文件连接
- **低心智负担**：对于简单的线性流程，这种架构开发最快

### 1.3 缺点（痛点）
| 缺陷 | 描述 |
|------|------|
| 状态丢失 | 如果图片生成到第 50 张断电了，Shell 脚本很难判断从哪里恢复 |
| 数据隔离差 | 脚本间共享全局环境变量或硬编码路径，移动项目文件夹就会报错 |
| 并发瓶颈 | Shell 很难优雅地处理多线程 API 请求 |
| 不可观测性 | llm_logger.py 目前是孤立的，没有生命周期钩子 |
| 硬编码隐患 | analyze_srt.py 中硬编码了 127.0.0.1:7890 代理 |

## 二、目标架构 (TO-BE)

### 架构模式：状态驱动的插件化编排架构 (State-Driven Orchestrator)

### 核心原则
1. **状态优先 (State-First)**：脚本运行前先读 metadata.json，运行后必写 metadata.json
2. **契约通信 (Data Contract)**：脚本间通过 JSON 交换产物路径与参数，而非物理文件命名
3. **环境隔离 (Environment Sandbox)**：网络代理、API Key、路径变量统一由底座加载

### 目标架构图
```
graph TD
    subgraph "Layer 1: 底座与配置 (Ground)"
        A[config.sh / .env] --> B(环境自适应 & 代理检查)
        B --> C{Guard: 准入校验}
    end

    subgraph "Layer 2: 编排器与记忆 (Brain)"
        C -- Pass --> D[produce_mv.sh / manager.py]
        D <--> E[(metadata.json 状态库)]
        D -. 监控 .-> F[llm_logger.py]
    end

    subgraph "Layer 3: 原子技能插件 (Skills)"
        D --> G[Music Gen]
        D --> H[Lyrics Gen]
        D --> I[SRT Analyze]
        D --> J[Image API]
        D --> K[FFmpeg Assemble]
    end

    E -. 进度反馈 .-> D
    J -- 重试机制 --> J
```

## 三、架构优化方向

| 维度 | 目前架构 | 优化后架构 |
|------|----------|------------|
| 执行逻辑 | A -> B -> C (死板) | 任务调度器根据状态触发 (灵活) |
| 容错性 | 一处断电，全盘重来 | 支持断点续传、异常重试 |
| 扩展性 | 修改一个脚本可能影响全局 | 插件式替换，互不干扰 |
| 数据流 | 物理文件散落在各处 | 统一的 Context 内存映射 + 结构化存储 |
