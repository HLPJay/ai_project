可以把多个大模型当成一个“项目团队”，而不是每个问题都丢给最强模型。核心原则是：

**强模型负责方向、架构、关键决策；便宜/额度宽松模型负责执行、改写、批处理；你自己维护上下文摘要，避免每次重复喂大量 token。**

下面是一套比较省 token、适合做项目的工作流。

------

## 一、先给模型分工

你可以这样分：

### 1. ChatGPT / Claude：做“架构师 + 代码审查”

适合用在：

```
需求拆解
技术方案设计
数据库/目录结构设计
复杂 bug 分析
关键代码审查
安全性、可维护性评估
重构方案
```

不要频繁让它们做低价值重复劳动，比如“把这一堆日志全读完”“帮我生成 30 个差不多的组件”。

### 2. MiniMax / 其他模型：做“执行员 + 文案员”

适合用在：

```
批量生成页面文案
补注释
生成测试用例初稿
整理 changelog
翻译
润色 README
根据明确指令改小段代码
```

它们的任务最好是**边界清晰、输入少、输出格式固定**。

### 3. 本地工具 / IDE / 脚本：做“机械活”

比如：

```
搜索代码
批量替换
格式化
跑测试
找大文件
统计目录
生成依赖树
```

这些不要浪费模型 token。能用 `grep`、`ripgrep`、`git diff`、`pytest`、`npm test` 解决的，就别问大模型。

------

## 二、建立一个“项目上下文文档”

这是省 token 最关键的一步。

你创建一个文件，比如：

```
PROJECT_CONTEXT.md
```

里面固定写：

```
# 项目目标
我要做什么，最终交付是什么。

# 技术栈
前端：
后端：
数据库：
部署方式：

# 当前目录结构
只写关键目录，不要贴完整 node_modules。

# 已完成
- ...

# 当前问题
- ...

# 关键约定
- 不提交 node_modules
- 使用 pnpm / npm
- 使用 TypeScript
- API 返回格式为 ...

# 最近一次决策
- 为什么这样设计
- 哪些方案被否决
```

以后每次问模型，不要把整个项目重新贴一遍，只贴：

```
请基于以下项目上下文 + 当前问题回答：
【PROJECT_CONTEXT 摘要】
【当前报错 / 当前代码片段 / 当前目标】
```

这样能把每次请求从几万 token 降到几百到几千 token。

------

## 三、每次提问都用“最小上下文包”

不要这样问：

```
这是我整个项目，你帮我看看怎么改。
```

要这样问：

```
目标：修复 git push 被大文件拒绝的问题。
当前状态：
- 分支 dev
- ahead origin/dev by 2 commits
- GitHub 拒绝 node_modules/.remotion/chrome-headless-shell.exe
- git status clean
我已经执行过：
- git rm -r --cached xxx，提示 pathspec not match
请给我最安全的修复步骤，不要解释基础概念。
```

这样模型不需要重新理解全局，只解决当前问题。

------

## 四、推荐的项目推进流程

### 第 1 步：用强模型做总体规划

一次性问 ChatGPT 或 Claude：

```
我要做一个 XXX 项目，技术栈是 XXX。
请帮我拆成：
1. MVP 功能
2. 目录结构
3. 数据模型
4. 开发里程碑
5. 每个阶段的验收标准
要求尽量可落地，不要过度设计。
```

拿到结果后，保存到：

```
PROJECT_PLAN.md
```

以后不要重复问“我应该怎么做”，只问某个阶段的具体任务。

------

### 第 2 步：让强模型生成任务清单

比如：

```
基于这个项目计划，把第一阶段拆成 10 个可以逐个提交的 Git 任务。
每个任务包含：
- 目标
- 涉及文件
- 修改点
- 验收命令
```

这一步很省后续 token，因为你之后每次只执行一个小任务。

------

### 第 3 步：每次只让模型处理一个小任务

例如：

```
当前任务：给 Remotion demo 增加字幕时间轴组件。

上下文：
- 使用 React + Remotion
- 字幕数据格式：[{start, end, text}]
- 当前文件：src/Caption.tsx

请只输出需要新增/修改的代码，不要重写整个项目。
```

不要让模型一次性“完成整个系统”。大任务会导致上下文爆炸、反复返工、token 浪费。

------

### 第 4 步：执行后只把 diff 发给模型审查

非常重要。

不要把完整文件贴给模型，优先发：

```
git diff
```

然后问：

```
这是我刚完成的 diff。
请检查：
1. 是否有明显 bug
2. 是否会破坏现有逻辑
3. 有没有安全/性能问题
4. 是否可以提交
只指出必须修改的问题。
```

这比贴整个项目省很多 token，而且审查质量更高。

------

## 五、不同模型之间怎么配合

### 模式 A：一个强模型做主架构，另一个强模型做反方审查

比如：

1. 用 ChatGPT 生成方案。
2. 把方案摘要给 Claude。
3. 问 Claude：

```
请作为代码审查者，指出这个方案最可能失败的 5 个点。
不要重写方案，只提风险和修正建议。
```

这样比让两个模型都完整做一遍省 token。

------

### 模式 B：强模型给指令，弱/便宜模型执行

比如先让强模型输出明确任务：

```
请把这个任务写成给初级开发者执行的 checklist。
```

然后把 checklist 给 MiniMax：

```
请按 checklist 生成 README 初稿。
```

或者：

```
请根据这个接口定义生成 5 个测试样例。
```

------

### 模式 C：弱模型先整理，强模型后判断

如果你有很多日志、报错、文档，可以先让额度宽松的模型做摘要：

```
请把这段日志压缩成：
1. 关键错误
2. 涉及文件
3. 已尝试操作
4. 仍未解决的问题
控制在 300 字以内。
```

然后把 300 字摘要给强模型分析。

## 六、控制 token 的具体技巧

### 1. 永远不要贴无关文件

例如问 Git 问题，只需要：

```
git status
git log --oneline -5
报错全文
你执行过的命令
```

不需要贴项目代码。

问某个函数 bug，只贴：

```
相关函数
调用方式
报错堆栈
预期行为
实际行为
```

------

### 2. 让模型先问你要哪些文件

可以这样：

```
我想修这个 bug。请先告诉我你需要看哪些文件，不要直接猜。
```

然后你只贴它要的 2-3 个文件片段。

------

### 3. 让模型输出 patch，而不是完整文件

用这个提示词：

```
请只输出最小修改方案。
格式：
1. 修改哪个文件
2. 替换哪段代码
3. 新代码
不要输出完整文件。
```

这样最省。

------

### 4. 用摘要替代历史对话

每完成一个阶段，让模型生成一次：

```
请把本轮讨论压缩成项目状态摘要，供下一次对话使用。
包括：
- 当前目标
- 已完成修改
- 当前问题
- 关键决策
- 下一步
控制在 500 字以内。
```

保存下来。下一次新会话直接贴摘要，不贴全部历史。

------

### 5. 不要让模型“顺便优化”

“顺便看看有没有问题”很耗 token，也容易引入不必要改动。

更好的问法：

```
只检查会导致运行失败的问题。
不要做风格优化。
不要改无关代码。
```

------

## 七、一个适合你的日常工作流

你可以按这个循环来：

```
1. 人类决定当前目标
2. 强模型拆任务
3. 自己/IDE执行机械操作
4. 小模型生成样板代码或文案
5. 强模型审查 git diff
6. 本地跑测试
7. 提交
8. 更新 PROJECT_CONTEXT.md
```

每个任务控制在一个 commit。

比如：

```
git checkout -b fix/remotion-vad
# 修改代码
npm test
git diff
# 发 diff 给模型审查
git commit -m "fix: disable vad after demucs separation"
```

## 八、推荐你建立 4 个固定提示词

### 1. 架构提示词

```
你是项目架构审查员。
请基于当前上下文给出最小可行方案。
要求：
- 不过度设计
- 优先减少改动
- 明确涉及文件
- 明确验收标准
- 不要输出完整代码，除非我要求
```

### 2. Debug 提示词

```
你是资深 Debug 工程师。
请根据报错和我已尝试的命令判断根因。
要求：
- 先给最可能原因
- 再给最安全修复步骤
- 每一步说明执行目录
- 不要建议删除整个项目
- 不要省略风险提示
```

### 3. 代码审查提示词

```
你是代码审查员。
我会提供 git diff。
请只检查：
1. 会导致运行失败的问题
2. 数据丢失/安全风险
3. 明显逻辑错误
4. 是否符合本任务目标
不要进行风格优化。
```

### 4. 摘要提示词

```
请把这次对话压缩成下一次可继续使用的项目摘要。
包括：
- 项目目标
- 当前分支/状态
- 已完成事项
- 关键文件
- 未解决问题
- 下一步建议
控制在 500 字以内。
```

------

## 九、什么时候用最强模型

只在这些场景用：

```
需求不清，需要判断方向
架构设计
复杂 bug 多次失败
涉及数据安全/部署/支付/权限
大规模重构
生成最终方案
审查关键 diff
```

不要用最强模型做：

```
格式化 JSON
生成普通文案
翻译 README
批量注释
简单正则
普通 CRUD 样板代码
```

------

## 十、最省 token 的核心口诀

```
先规划，再拆小。
只贴相关，不贴全量。
先摘要，再提问。
让强模型决策，让便宜模型执行。
用 git diff 审查，不用全文件审查。
每轮结束更新项目上下文。
```

对于你这种同时有 ChatGPT、Claude、MiniMax 的情况，我建议：

```
ChatGPT：主线规划、debug、工具链问题、代码修复策略
Claude：复杂代码审查、架构反思、长文档总结
MiniMax：文案、批量生成、简单代码、测试样例
本地工具：搜索、格式化、运行、验证
```

这样最容易把 token 花在“判断力”上，而不是浪费在重复阅读和机械输出上。
