第 3 项“补单元测试”最适合用 **强模型设计测试范围，小模型生成样板，你本地运行，强模型审查失败原因** 的流程。核心是：**不要让模型一次性给整个项目补测试**，而是按模块、按行为、按 commit 控制。

## 推荐流程

### 第一步：让强模型只设计测试清单

先不要写代码。给 Claude 或 ChatGPT：

```
任务：为 config_manager.py 设计单元测试清单。
要求：
1. 不写测试代码
2. 只列测试场景
3. 每个场景说明输入、预期输出、需要 mock 的内容
4. 优先覆盖高风险逻辑
5. 控制在 10 个以内
```

这样用 token 很少，而且能避免一上来生成一大堆跑不通的测试。

------

### 第二步：你挑 3～5 个最关键场景

比如 `config_manager.py`，优先测：

```
1. 默认配置是否正确加载
2. .env / 环境变量是否覆盖默认值
3. 缺失配置时是否返回默认值
4. bool/int 类型转换是否正确
5. 无效配置值是否安全处理
```

不要第一轮就追求 100% 覆盖率。先做能防止核心逻辑坏掉的测试。

------

### 第三步：让 MiniMax 写测试初稿

把**单个模块代码 + 测试清单**给 MiniMax，不要给全项目。

提示词：

```
请根据下面的 config_manager.py 和测试清单，生成 pytest 单元测试。
要求：
1. 只测试公开接口
2. 不访问真实 .env
3. 使用 tmp_path / monkeypatch
4. 不依赖真实 API token
5. 每个测试函数名称表达清楚行为
6. 只输出 tests/test_config_manager.py
```

MiniMax 适合做这类“明确输入 → 生成样板”的活。

------

### 第四步：你本地运行测试

```
pytest tests/test_config_manager.py -q
```

失败后不要把所有项目文件都发给模型，只发：

```
1. 失败的测试函数
2. 报错堆栈
3. 相关函数代码
4. 你期望的行为
```

比如：

```
下面是 pytest 失败信息。
请只修测试，不要改业务代码，除非业务代码确实有 bug。
```

------

### 第五步：用强模型审查测试质量

等测试跑通后，把 `git diff` 给 ChatGPT 或 Claude：

```
这是新增的测试 diff。
请审查：
1. 是否真的覆盖了核心行为
2. 是否有无意义测试
3. 是否依赖本机环境
4. 是否会因为执行顺序不同而失败
5. 是否需要补一个边界用例
不要重写全部代码，只指出必须修改的问题。
```

## 三个模块怎么排优先级

你现在报告里提到：

```
config_manager.py
align.py
log_setup.py
```

建议顺序是：

```
1. config_manager.py
2. log_setup.py
3. align.py
```

原因：

**`config_manager.py`**
 优先级最高。配置错了，全流程都会错，而且测试通常容易 mock。

**`log_setup.py`**
 改动小，适合快速建立测试基础。比如日志文件创建、日志级别、轮转 handler 是否使用正确。

**`align.py`**
 最复杂，不要一开始测完整 `run()`。先测纯函数和小算法。

------

## `align.py` 应该怎么控测试范围

不要直接测试：

```
LyricsAligner().run(...)
```

因为它会牵涉：

```
音频文件
Demucs
Whisper
faster-whisper
外部进程
缓存
真实项目目录
```

这会让测试又慢又脆弱。

优先测这些纯函数：

```
format_srt_time()
parse_srt_time()
parse_lyrics()
SimilarityScorer.similarity()
SimilarityScorer.chinese_overlap()
SimilarityScorer.score_pair()
LyricsAligner._repair_alignment_timeline()
LyricsAligner._alignment_timeline_is_suspicious()
LyricsAligner._apply_uniform_timeline()
```

这些不需要真实音频，不需要 API，不需要 GPU，最省成本。

给模型的任务卡可以这样写：

```
任务：为 align.py 增加第一批 pytest。
只测试纯函数，不测试 Demucs、Whisper、run()。
范围：
- format_srt_time
- parse_srt_time
- SimilarityScorer.chinese_overlap
- LyricsAligner._repair_alignment_timeline
要求：
1. 不需要真实音频
2. 不启动外部进程
3. 不 import 大模型
4. 测试运行时间 < 1 秒
5. 每个测试只验证一个行为
```

------

## 每一轮测试任务的大小

每一轮只做一个 commit：

```
commit 1: test config_manager core config lookup
commit 2: test log setup handler creation
commit 3: test align timestamp helpers
commit 4: test align timeline repair
commit 5: test scene generator prompt building if needed
```

不要一个 commit 塞：

```
config_manager + align + pipeline + scene_generator + README
```

否则测试失败时很难定位。

------

## 如何控制 token 使用

你给模型的上下文只需要这几样：

```
1. 当前目标
2. 被测模块的相关代码
3. 测试清单
4. 项目使用 pytest
5. 不能访问真实 API / 不能启动外部进程
```

不要发：

```
整个项目
所有 README
完整日志
node_modules
无关文件
```

如果文件很长，比如 `align.py`，你可以只贴要测的函数。比如只贴：

```
format_srt_time
parse_srt_time
SimilarityScorer
_repair_alignment_timeline
```

## 推荐你固定一个“测试生成提示词”

```
你是 Python 测试工程师。
请为下面代码生成 pytest 单元测试。

限制：
1. 只测试我指定的函数
2. 不访问网络
3. 不调用真实 API
4. 不启动外部命令
5. 不依赖本机绝对路径
6. 使用 tmp_path、monkeypatch、caplog 等 pytest 工具
7. 不为了测试而修改业务代码
8. 如果代码不可测试，先说明需要的最小重构点

输出：
- 只输出测试文件内容
- 不解释
```

## 推荐验收标准

每补一个模块测试，都用这三个命令验收：

```
pytest tests/test_config_manager.py -q
pytest tests/test_log_setup.py -q
pytest tests/test_align.py -q
```

如果你有覆盖率工具：

```
pytest tests/test_config_manager.py --cov=src.config_manager --cov-report=term-missing
```

不要一开始追求全项目覆盖率。先追求：

```
核心纯逻辑有测试
测试稳定
不依赖真实环境
失败信息清楚
每个测试能在 1 秒内跑完
```
