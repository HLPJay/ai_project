# 知识库查看器写入功能实现计划

## 上下文
当前知识库查看器是一个只读系统，用户希望添加完整的CRUD（创建、读取、更新、删除）功能，包括文件和目录操作。用户要求尽量不改动原有文件，通过新增文件的方式实现功能。

## 现状分析
- 后端：`server.js` - Node.js HTTP服务器，提供只读API
- 前端：`public/index.html` - 单页面应用，Markdown渲染查看器
- API端点：`/api/tree`, `/api/file`, `/api/search`, `/api/image`（均为GET）
- 无用户认证，无写入功能

## 设计目标
1. 添加完整的CRUD操作：创建、读取、更新、删除文件和目录
2. 添加Markdown编辑器页面，支持双栏编辑（源码+预览）
3. 保持原有代码最小改动，主要通过新增文件实现

## 技术方案

### 1. 后端API扩展（最小化修改server.js）
在`server.js`中添加对POST、PUT、DELETE方法的支持，将CRUD请求路由到新的处理模块。

**新增文件**：`crud-api.js`
**server.js修改**：添加3个条件分支（约10行代码）

#### API设计
- **POST /api/files** - 创建新文件
  - 参数：`path`（文件路径）, `content`（文件内容）
  - 返回：创建的文件信息

- **PUT /api/file** - 更新现有文件（扩展现有端点）
  - 参数：`id`（base64编码的文件路径）, `content`（新内容）
  - 返回：更新后的文件信息

- **DELETE /api/file** - 删除文件
  - 参数：`id`（base64编码的文件路径）
  - 返回：删除状态

- **POST /api/directories** - 创建目录
  - 参数：`path`（目录路径）
  - 返回：创建的目录信息

### 2. 前端编辑器页面
**新增文件**：`public/editor.html`

#### 功能特性
- 双栏布局：左侧Markdown源码编辑器（textarea），右侧实时预览
- 实时预览：使用marked.js将Markdown转换为HTML
- 文件操作：保存、取消、删除（编辑时）、新建
- 路径输入：新建文件时指定路径和文件名
- 从主页面跳转：通过URL参数`?id=...`传递要编辑的文件ID

#### UI组件
1. **工具栏**：保存按钮、取消按钮、删除按钮（仅编辑时）、新建按钮
2. **编辑器区域**：左右分栏，可调整宽度
3. **文件信息栏**：显示当前文件路径、保存状态
4. **路径输入框**：新建文件时可见

### 3. 主页面集成
**index.html修改**：添加"编辑"按钮（约5行代码）
- 在文件列表项添加"编辑"按钮
- 在工具栏添加"新建文件"按钮
- 点击后跳转到`editor.html`页面

## 实施步骤

### 阶段1：后端API扩展

### 阶段3：主页面集成
1. 在`index.html`中添加编辑和新建按钮
2. 实现到编辑器页面的跳转逻辑
3. 添加URL参数传递支持

### 阶段4：测试与优化
1. 测试所有CRUD操作
2. 测试错误处理（网络错误、权限错误等）
3. 添加用户反馈（加载状态、成功/错误提示）
4. 优化编辑器体验（自动保存草稿、快捷键等）

## 安全考虑
1. **路径安全检查**：确保所有文件操作限制在`NOTES_DIR`目录内
2. **输入验证**：验证文件名和路径，防止目录遍历攻击
3. **错误处理**：友好的错误提示，不暴露内部路径信息
4. **CORS设置**：保持现有的CORS设置

## 验证方案
1. 启动服务：`node server.js`
2. 访问主页面：`http://localhost:3000`
3. 点击"新建文件"按钮，跳转到编辑器
4. 创建新文件并保存
5. 返回主页面，确认文件出现在列表中
6. 点击文件旁边的"编辑"按钮，修改内容并保存
7. 测试删除功能
8. 测试目录创建功能

## 扩展可能性
1. **自动保存**：添加草稿自动保存功能
2. **版本历史**：简单的文件版本控制
3. **批量操作**：批量删除、移动文件

## 风险与缓解
1. **并发写入冲突**：简单实现，不处理并发（适合个人使用）
2. **数据丢失**：添加确认对话框，重要操作需确认
3. **浏览器兼容性**：使用现代JavaScript特性，支持主流浏览器
