方案一：/api/image 端点架构原理
1. 核心设计思想
    创建一个专用的 API 端点来安全地提供 NOTES_DIR 目录下的图片文件，而不是直接暴露整个目录作为静态文件服务。

2. 端点的角色与定位
    专用服务：专门处理图片资源请求
    安全代理：在文件系统路径和 HTTP 响应之间建立受控桥梁
    统一接口：保持与现有 /api/tree、/api/file 一致的 API 设计风格
3. 完整请求响应流程
前端处理阶段
    Markdown 解析：前端获取 Markdown 内容后，渲染引擎（如 marked.js）开始解析
    图片标签识别：解析器遇到 ![](path/to/image.png) 语法时创建 <img> 标签
    路径转换：
    相对路径（如 ./images/photo.png）需要转换为绝对路径
    路径编码：将文件在 NOTES_DIR 中的相对位置编码为 Base64 URL-safe 格式
    生成新 URL：/api/image?id=<encoded-path>
后端处理阶段
    请求接收：服务器收到 /api/image?id=... 请求
    参数解码：
        从查询参数 id 提取编码后的路径
        Base64 URL-safe 解码得到原始相对路径
    安全检查：
        路径规范化：防止 ../../../ 目录遍历攻击
        边界验证：确保解析后的绝对路径以 NOTES_DIR 为前缀
        文件存在性检查：验证目标文件确实存在
    MIME 类型识别：
        根据文件扩展名（.png, .jpg, .gif 等）确定 Content-Type
        复用现有的 mime() 函数扩展图片支持
    数据流传输：
        以二进制流模式读取图片文件
        设置正确的 Content-Type 和缓存头
        将文件内容流式传输到 HTTP 响应
4. 安全机制原理
路径安全
    编码防御：Base64 编码防止特殊字符注入
    路径规范化：path.resolve() 消除 . 和 .. 符号
    目录边界检查：startsWith(NOTES_DIR) 确保文件在允许范围内
文件类型控制
    扩展名白名单：只允许特定图片格式（.png, .jpg, .jpeg, .gif, .svg, .webp）
    MIME 验证：文件内容与扩展名的一致性检查（可选增强）
    访问控制
    仅读权限：端点只支持 GET 请求
    无目录列表：不能通过此端点浏览目录结构
5. 性能优化原理
缓存策略
    Last-Modified：基于文件修改时间
    ETag：文件内容哈希值
    Cache-Control：浏览器缓存指令
传输优化
    流式传输：大文件不一次性加载到内存
    范围请求支持：可支持 HTTP Range 请求（断点续传）
6. 与现有架构的集成点
复用现有组件
    路径处理：复用 /api/file 的路径解码和安全检查逻辑
    错误处理：复用 notFound() 和错误响应模式
    MIME 类型：扩展现有的 mime() 函数
API 设计一致性
    参数格式：使用相同的 id 参数和 Base64 编码
    响应格式：统一的错误响应结构
    CORS 策略：保持相同的跨域配置
7. 前端适配需求
Markdown 渲染层适配
    预处理策略：在 Markdown 渲染前转换所有图片链接
    正则匹配：识别 ![]() 语法中的路径
    路径计算：基于当前文件位置计算图片的相对路径
备选方案：DOM 后处理
    选择器定位：渲染后查找所有 <img> 标签
    属性替换：将 src 属性转换为 API 格式
    懒加载支持：可集成 Intersection Observer 实现懒加载
8. 扩展性考虑
    功能扩展点
    图片预处理：添加查询参数支持缩略图生成（?width=200&height=200）
    格式转换：支持 WebP 自动转换
    水印添加：可选的水印功能
    EXIF 信息：返回图片的元数据信息
    监控与日志
    访问统计：记录图片请求频率
    性能监控：跟踪图片加载时间
    错误追踪：记录文件缺失等错误情况
9. 优缺点分析
优点
    安全性高：细粒度的路径控制，无目录遍历风险
    架构一致：与现有 API 设计模式统一
    功能可控：可轻松添加图片处理功能
    缓存友好：可针对图片特性优化缓存策略
缺点
    前端修改：需要调整 Markdown 渲染流程
    编码开销：路径编码/解码的额外处理
    单点服务：所有图片请求都经过同一端点
这个方案的核心价值在于：在不暴露文件系统结构的前提下，提供安全、可控的图片访问服务，同时保持系统架构的一致性和可扩展性。
